# v1.8.1 — 실행 안정성 개선

v1.8.1은 게임 기능보다 오케스트레이션의 안전성, 재현성과 관측 가능성을 강화한다.
2.5D 횡스크롤 게임 제작 실증에서 확인된 다음 문제를 기준으로 설계했다.

- `Agent.run_turn()` 밖에서 직접 호출한 MCP 도구가 v1.8 로그에 남지 않음
- 검증 요청에서도 모델이 파일과 씬을 다시 만들려고 시도함
- 마일스톤 재시도가 이미 성공한 산출물을 다시 설치하거나 덮어씀
- `unity_add_component` 반복 호출로 컴포넌트가 중복됨
- 플레이 모드 편집 명령이 실패를 반환하기 전에 일부 값을 바꿈
- 스크린샷의 상위 폴더가 없으면 저장 실패
- `tap` 직후 응답만으로 자동 해제 완료 여부를 알 수 없음
- CP949 콘솔 콜백이 일부 Unicode 문자를 출력하다 에이전트를 중단함

## MCP 호출 감사 로그

`UnityTools.call()` 경계에서 모든 도구 호출을 JSONL로 기록한다. 따라서 CLI와
`Agent.run_turn()`뿐 아니라 테스트 및 별도 Python 코드의 직접 호출도 포함된다.

```text
logs/mcp/YYYY/MM/DD/
  YYYYMMDD_HHMMSS_mmm_unity_tools_<session-id>.jsonl
```

주요 이벤트는 `session_started`, `tool_mode_changed`, `tool_call_started`,
`tool_call_finished`, `session_finished`이다. 도구의 전체 인자·결과와 경과 시간이
기록된다. 로그 I/O 실패는 Unity 작업을 차단하지 않는다.

| 환경변수 | 기본값 | 설명 |
|---|---|---|
| `UNITY_MCP_AUDIT_LOGS` | `1` | `0`이면 MCP 감사 로그 비활성화 |
| `UNITY_MCP_AUDIT_LOG_DIR` | `logs/mcp` | 감사 로그 루트 경로 |

## 검증 전용 모드

CLI에서 다음처럼 실행한다.

```text
/verify 현재 씬의 이동, 점프, 카메라 추적과 오류 0건을 확인해줘
```

이 모드는 도구 스키마에서 편집·파일 쓰기 도구를 제거하고, 누수된 tool-call이
도구 이름을 직접 호출해도 `UnityTools.call()`에서 다시 차단한다. 허용 범위는 상태,
계층, 콘솔, 플레이 모드, 입력, 대기, 스크린샷과 읽기 도구다. 한 요청이 끝나면 이전
도구 모드로 자동 복원된다. 환경변수 `UNITY_AGENT_TOOL_MODE=verify`로 전체 세션을
검증 전용으로 시작할 수도 있다. 모델이 호출 한도나 오류로 조기 종료되더라도 검증 중
직접 진입한 플레이 모드는 호스트가 `finally` 단계에서 자동 종료한다.

## 브리지 안전성

- `unity_add_component`: 기본 멱등. 기존 컴포넌트면 `alreadyPresent=true`
- `unity_remove_component`: 첫 컴포넌트 또는 `remove_all=true`로 중복 전체 제거
- 모든 씬/컴포넌트/머티리얼 편집: 플레이 모드 검사 후에만 변경 시작
- `unity_modify_gameobject`: 새 parent 존재 여부를 변경 전에 검증
- `unity_screenshot`: 누락된 상위 폴더 자동 생성
- `unity_get_input_state`: 현재 held 키와 예약된 tap release 조회

## 마일스톤 재시도

재시도 프롬프트는 산출물 대장의 파일·씬·오브젝트를 이미 성공한 결과로 명시한다.
모델은 먼저 기존 상태를 조회하고, 실제 누락이나 오류가 없으면 설치·생성·덮어쓰기를
반복하지 않고 미완료 검증부터 이어가도록 지시받는다. 컴포넌트 계층에서는 브리지의
멱등 처리도 중복을 결정적으로 방지한다.

## Windows 출력 안전성

CLI의 표준 입력·출력·오류를 UTF-8 `errors="replace"`로 설정한다. 임베딩 코드가
CP949 같은 제한된 콜백을 전달해도 Unicode 출력 실패는 UI 표시만 생략하고 실제
Unity 작업과 UTF-8 로그 기록은 계속된다.
