# v1.4.1 — read_console stale 에러 수정

2026-07-06. v1.4 삭제 기능이 "안 먹히는" 것처럼 보이던 문제의 진짜 원인 해결.

## 문제

v1.4에서 `unity_delete_script`를 추가했는데도 "기존 cs 파일이 삭제되지 않는" 것처럼 보였다.
실제로는 **삭제는 정상 작동**했고(파일은 디스크에서 사라짐), `unity_read_console`가 **이미
삭제된 파일의 옛 컴파일 에러를 계속 반환**하는 게 문제였다. 모델은 콘솔이 여전히 에러를
보고하니 "삭제가 실패했다"고 오판하고, 없는 파일을 무한히 삭제 시도하다 반복 한도에 걸렸다.

증상 로그: `unity_delete_script`가 `file not found`를 반환(=이미 없음)하는데도
`unity_read_console`는 `TetrominoBlock.cs(55,17): CS0111` 에러를 계속 반환. 라이브 확인 시
`isCompiling:false`인데 콘솔 에러 5건이 전부 타임스탬프 **15:37:21**로 고정 — 명백한 stale.

## 메커니즘: 로그 버퍼가 도메인 리로드에만 의존

브리지의 `logBuffer`([UnityMcpBridge.cs](../../My%20project/Assets/Editor/McpBridge/UnityMcpBridge.cs))는
`Application.logMessageReceivedThreaded`로 쌓기만 하는 append-only 버퍼다. 이걸 비우는 유일한
계기는 **도메인 리로드**(정적 필드 재초기화)뿐인데, 도메인 리로드는 **컴파일 성공 시에만**
일어난다.

그런데 프로젝트에 실패 세션들이 남긴 **실제 컴파일 에러**(`Tetromino` 클래스가 `Tetromino.cs`와
`PlayerController.cs`에 이중 정의, CS0101)가 있어 컴파일이 계속 실패했다.
→ 도메인 리로드 없음 → 이미 삭제된 `TetrominoBlock.cs`의 옛 에러가 버퍼에서 영영 안 지워짐.

즉 **컴파일이 깨진 동안에는 read_console가 삭제·수정된 파일의 stale 에러를 무한정 반환**한다.
하필 이 상황이 v1.4 삭제 기능이 가장 필요한 순간이라, 삭제가 무력해 보였다.

## 해결 1 (핵심): 컴파일 시작 시 버퍼 비우기

`UnityMcpBridge.cs` — `using UnityEditor.Compilation;` 추가, 정적 생성자에 등록:

```csharp
CompilationPipeline.compilationStarted -= OnCompilationStarted;
CompilationPipeline.compilationStarted += OnCompilationStarted;

static void OnCompilationStarted(object context)
{
    lock (logLock) logBuffer.Clear();
}
```

컴파일이 **실패해 도메인 리로드가 없어도**, 매 컴파일 시작 시 버퍼를 비운다. 따라서
read_console는 항상 '이번 컴파일'의 fresh 에러만 반환하고, 삭제된 파일의 stale 에러는 사라진다.
Unity 콘솔의 "Clear on Recompile" 토글과 동일한 동작. 생성자는 도메인 리로드마다 재실행되며
`-=`/`+=`로 중복 등록을 막는다(기존 패턴).

## 해결 2 (배포 + 클린슬레이트): 테트리스 스크립트 정리

**닭-달걀**: 브리지 C# 변경을 반영하려면 프로젝트가 한 번은 컴파일에 성공해야 하는데(도메인
리로드로 새 어셈블리 로드), 중복 `Tetromino`로 불가능했다. 사용자 선택에 따라 실패 세션 잔재를
정리해 컴파일을 통과시켰다.

- 전체 `Assets/Scripts` 백업 후, 테트리스 클러스터 10개 삭제(`unity_delete_script`로 도그푸딩):
  `TetrominoBlockController`, `TetrisGame`, `TetrisGameManager`, `TetrisController`,
  `TetrisGameSimple`, `Tetromino`, `GameBoard`, `TetrisMatrix`, 그리고 이들과 결합돼 단독
  컴파일 불가한 `GameManager`(→GameBoard), `PlayerController`(currentTetromino.blocks 사용 +
  잉여 Tetromino 클래스).
- 유지: `BallController`, `BallMovement`, `SphereMovement`, `PlayerMovement`(테트리스 무참조).
- refresh → 컴파일 성공 → 도메인 리로드 → 새 브리지(해결 1) 로드 + stale 버퍼 초기화.

> 컴파일 중 남은 Ball/Sphere 스크립트의 구식 API가 "Script Updating Consent" 모달을 띄웠다
> (v1.2 이슈). raw 소켓 헬퍼는 모달 자동 해제를 안 거쳐 잠시 멈췄고, 모달 처리 후 컴파일이
> 완료됐다. 실사용 경로(`mcp_client`)는 `_dismiss_known_modals`로 자동 처리한다.
