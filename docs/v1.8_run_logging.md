# v1.8 — 요청별 실행 로그

## 결론: v1.7에는 실행 로그가 없었다

v1.7까지 저장되던 `mcp_server.log`는 MCP 자식 프로세스의 stderr를 담는 진단
파일이다. 서버가 시작될 때 `w` 모드로 열려 이전 내용이 덮어써지며, 다음 정보는
보존하지 않았다.

- 사용자 요청 원문
- 플래너가 만든 마일스톤
- 모델의 전체 응답과 tool-call
- tool 인자와 결과
- TaskContract의 검증 누락 및 정책 차단
- 계획의 성공·실패·미착수 상태
- 예외 또는 사용자 중단 원인

따라서 v1.7의 `mcp_server.log`를 실행 트랜스크립트로 볼 수 없다. v1.8에서 별도의
요청별 로그 계층을 추가했고, v1.8.1에서 Agent 밖의 직접 MCP 호출까지 감사 로그로
확장했다.

## 저장 단위와 위치

CLI에서 사용자 요청 하나를 입력하거나 외부 코드가 `Agent.run_turn()`을 한 번
호출할 때마다 로그 파일 두 개가 생성된다. 플래너 사용 여부와 무관하다.

```text
logs/runs/YYYY/MM/DD/
  YYYYMMDD_HHMMSS_mmm_<요청요약>_<run-id>.log
  YYYYMMDD_HHMMSS_mmm_<요청요약>_<run-id>.jsonl
```

동일한 timestamp와 run-id를 가진 두 파일이 한 실행을 나타낸다. 파일 생성은
exclusive 모드를 사용하므로 기존 로그를 덮어쓰지 않는다.

## 파일 형식

### `.log`

사람이 문제를 조사하기 위한 UTF-8 텍스트 파일이다. 각 이벤트는 시각과 이벤트
이름, 들여쓰기된 JSON payload로 구성된다.

### `.jsonl`

한 줄에 JSON 오브젝트 하나가 기록된다. 자동 분석, 통계와 회귀 비교에 사용한다.
모든 레코드에 다음 공통 필드가 있다.

| 필드 | 의미 |
|---|---|
| `timestamp` | 로컬 timezone을 포함한 ISO-8601 시각 |
| `run_id` | 요청 실행을 식별하는 임의 ID |
| `event` | 이벤트 종류 |

## 주요 이벤트

| 이벤트 | 기록 내용 |
|---|---|
| `run_started` | 사용자 요청 원문과 모델 |
| `run_configuration` | planner/iteration 설정 |
| `execution_mode` | `single` 또는 `plan`, 정규화된 마일스톤 |
| `milestone_started` | 번호, 전체 개수와 제목 |
| `milestone_finished` | 성공/실패, 재시도 횟수와 남은 전체 예산 |
| `milestone_ping` | 마일스톤 경계 Unity 연결 결과 |
| `assistant_response` | 모델 전체 텍스트와 복구된 tool-call |
| `tool_result` | tool 이름, 전체 인자와 전체 결과 |
| `verification_incomplete` | 완료를 차단한 검증 항목 |
| `warning` | 정책 차단, 재시도와 루프 가드 경고 |
| `vision_result` | 로컬 비전 모델 결과와 이미지 경로 |
| `plan_report` | 완료/실패/미착수 마일스톤과 산출물 보고 |
| `exception` | 예외 종류, 메시지와 traceback |
| `run_finished` | 최종 outcome과 경과 시간 |

`run_finished.outcome`은 다음 중 하나다.

- `completed`: 계약과 산출물 검증을 통과
- `failed`: iteration 한도 또는 마일스톤 최종 실패
- `error`: 처리되지 않은 예외
- `interrupted`: KeyboardInterrupt 또는 asyncio cancellation

## 설정

| 환경변수 | 기본값 | 설명 |
|---|---|---|
| `UNITY_AGENT_RUN_LOGS` | `1` | `0`이면 요청 로그 비활성화 |
| `UNITY_AGENT_RUN_LOG_DIR` | `<저장소>/logs/runs` | 로그 루트 절대/상대 경로 |

예시:

```powershell
$env:UNITY_AGENT_RUN_LOGS="1"
$env:UNITY_AGENT_RUN_LOG_DIR="D:\unity-agent-logs"
uv run main.py
```

CLI는 요청이 끝날 때 두 로그 경로를 표시한다. `/log`를 입력하면 마지막 실행의
경로를 다시 확인할 수 있다.

## 적용 범위

로깅은 `main.py`에만 있지 않고 `Agent.run_turn()`에 있다. 따라서 다음 경로가 모두
자동 기록된다.

- 대화형 `uv run main.py`
- 플래너가 발동한 다중 마일스톤 요청
- 플래너가 발동하지 않은 단일 요청
- `e2e_v17_run.py`
- 별도 Python 코드에서 직접 호출한 `Agent.run_turn()`

`Agent.run_turn()`을 거치지 않고 `UnityTools.call()`을 직접 사용한 호출은 v1.8.1의
`logs/mcp/YYYY/MM/DD/*.jsonl`에 기록된다. 상세:
[v1.8.1 실행 안정성 개선](v1.8.1_reliability.md)

`/tools`, `/model`, `/reset` 같은 CLI 자체 명령은 Unity 작업 turn이 아니므로 요청별
실행 로그를 만들지 않는다. 그 명령에서 실제 MCP 도구가 호출된다면 MCP 감사 로그에는
기록된다.

## 장애 격리

로그 폴더 생성이나 파일 쓰기가 실패해도 Unity 작업은 중단하지 않는다. CLI에
경고를 표시하고 에이전트 실행은 계속한다. 로그 파일은 append-only로 기록되므로
프로세스가 강제 종료되어도 종료 직전까지 flush된 JSONL 이벤트가 남는다.

## 보안과 Git 정책

실행 로그에는 다음 민감할 수 있는 정보가 포함된다.

- 사용자의 전체 요청
- 생성·수정한 C# 및 JSON 원문
- 로컬 파일 절대경로
- Unity tool의 전체 응답

따라서 `logs/`는 `.gitignore`에 포함하며 기본적으로 GitHub에 올리지 않는다. 로그를
공유할 때는 경로, 프로젝트명과 코드 안의 자격 증명을 먼저 검토한다.

- `.log`와 `.jsonl` 동시 생성

- 로그 I/O 실패가 에이전트 작업을 차단하지 않는 장애 격리
