# v1.3 — 반복 오브젝트 생성으로 인한 턴 중단 해결 (배치 생성 + 전략 유도)

2026-07-06. 빈 씬에서 발견된 "너무 많은 툴 호출로 중단" 문제 해결.

## 문제

빈 씬에서 "10x20 매트릭스에 테트리스 구현"을 시키자, 모델(qwen3-coder:30b)이 스크립트는
정상 작성했으나 **200개(10×20) 셀을 `unity_create_gameobject` 호출 1회당 큐브 1개씩**
만들려다 `MAX_ITERS=15` 한도에 걸려 8개째에서 잘렸다. 모델은 "너무 많은 툴 호출로
중단되었습니다"라는 요약만 남기고 게임을 완성하지 못했다.

추가로, 그 과정에서 `scale`을 `"[0.9f, 0.9f, 0.9f]"`(C#풍 문자열)로 넘겨 pydantic 검증
에러가 나면서 반복 1회를 더 낭비했다.

### 진단: 모델 한계 + 하네스 설정 둘 다

- **모델 측**: 30B 로컬 모델은 "반복 생성은 스크립트 루프로 처리한다"는 상위 전략을 스스로
  선택하지 못하고, 눈앞의 툴을 그대로 N번 호출하는 경향이 있다.
- **하네스 측**: 그 대안 경로(배치 툴, 프롬프트 규칙)를 **전혀 제공하지 않았다.** 개별 생성
  툴 하나만 있으니 모델에게 다른 선택지가 없었다.

`MAX_ITERS`를 올리는 것은 해법이 아니다 — 200회 호출은 컨텍스트·시간상 어차피 불가능하고,
한도만 키우면 잘못된 전략이 더 오래 지속될 뿐이다. **해법은 전략 자체를 바꿔주는 것.**

## 해결: 5개 층

| Step | 파일 | 동작 | 구분 | 검증 |
|---|---|---|---|---|
| 1 | `UnityMcpBridge.cs` | `create_gameobjects` 배치 커맨드 (C#) | 핵심 | ✅ 라이브 브리지 스모크 |
| 2 | `unity_mcp/server.py` | `unity_create_gameobjects` MCP 툴 | 핵심 | ✅ 스키마 평탄성 + E2E |
| 3 | `agent.py` | 시스템 프롬프트 규칙 2줄 | 핵심 | ✅ 라이브 모델이 규칙 준수 |
| 4 | `mcp_client.py` | `_lenient_json` — `0.9f` 인자 자동 보정 | 선택 | ✅ 단위 테스트 |
| 5 | `agent.py` + `config.py` | 반복 호출 루프 가드 | 선택 | ✅ 단위 테스트 |

**설계 원칙**: Step 1·2가 대안 경로(배치 툴)를 만들고, Step 3이 모델을 그 경로로 유도하며,
Step 4·5는 모델이 그래도 실수/반복할 때의 안전망. 각 층이 독립적이라 하나가 실패해도 나머지가 받친다.

---

### Step 1 — C# 배치 생성 커맨드 (`UnityMcpBridge.cs`)

`Execute()` 디스패처에 `case "create_gameobjects"` 추가. 기존 `CmdCreateGameObject`의 본문
(primitive 파싱, 부모 해석 + 실패 시 `DestroyImmediate`, `ApplyTransform`, `Undo` 등록)을
공용 헬퍼 `CreateOne(JObject spec, out string error)`로 추출해 단일·배치 경로가 로직을 공유한다
(동작 불일치 방지).

`CmdCreateGameObjects` 핸들러 특성:
- `objects` 배열 필수, **최대 500개** (초과 시 "스크립트로 생성하라" 에러).
- **부분 실패 허용**: 항목별 에러를 index와 함께 수집(최대 20개), 실패해도 나머지는 계속 생성.
- **응답 압축**: `{created, failed, paths(앞 20개), errors(앞 20개)}` — 200개 경로 전체는
  토큰 낭비 + `TRUNCATE_CHARS=8000`에 어차피 잘림. 모델은 개수 확인 + 부모 경로만 있으면 충분.
- **배치 전체가 Ctrl+Z 한 번**: `Undo.IncrementCurrentGroup()` → 루프 → `CollapseUndoOperations()`.
- **순차 생성이라 배치 내 부모 참조 가능**: spec 0이 빈 `"Board"`, 이후 spec들이 `parent:"Board"`.
  이것이 바로 그리드/보드 패턴이다.
- 500개 `CreatePrimitive`도 에디터 메인 스레드 한 패스에서 수 ms — 청킹·비동기 불필요.

### Step 2 — Python MCP 툴 (`unity_mcp/server.py`)

`unity_create_gameobjects(objects: list[dict])` 추가. docstring이 **"3개 이상 유사 오브젝트는
반복 호출 대신 이걸 쓰라, 50개 이상은 스크립트 Awake()에서 생성하라"**를 명시.

- 타입을 `list[dict]`로 둬서 스키마를 **평평하게** 유지 (`mcp_client.py`의 `_simplify`가
  `$ref/$defs`를 인라인하지 못하므로 중첩 pydantic 모델을 쓰면 30B 모델이 스키마를 못 따름).
  사양 필드 문서는 docstring이 담당.
- 그리드 전용 모드는 넣지 않음 — 대량(200개) 사양을 모델이 JSON으로 생성하면 ~6–7k 토큰
  낭비 + 로컬 모델에서 truncation 위험. 큰 레이아웃은 Step 3 규칙으로 스크립트 경로로 유도하고,
  배치 툴은 3~50개 구간을 담당하게 역할 분담.
- `mcp_client.py`가 시작 시마다 server.py를 새로 spawn하고 `list_tools()`로 발견하므로 추가
  등록 작업 없음.

> **참고**: server.py는 그동안 "수정 금지"로 취급했으나, 배치 기능은 브리지(C#)와 서버(Python)
> 양쪽에 손을 대야만 성립하므로 이번에 사용자 승인 하에 수정했다.

### Step 3 — 시스템 프롬프트 규칙 (`agent.py`)

`Rules:` 블록에 2줄 추가:

1. **엄격한 JSON 인자** — `0.9f` 같은 C# 접미사 금지, 벡터는 진짜 JSON 배열
   (`[0.9, 0.9, 0.9]`)이지 문자열 `"[0.9f, ...]"`가 아님.
2. **반복 생성 금지** — 3개 이상 유사 오브젝트는 `unity_create_gameobjects` 1회, 큰/절차적
   레이아웃(그리드·보드, 20개 이상 — 예: 10×20 보드)은 **아예 툴로 만들지 말고** MonoBehaviour의
   `Awake()`/`Start()`에서 `CreatePrimitive`/`Instantiate` 루프로 생성해 게임이 스스로 그리게 함
   (툴 호출 0회).

`MAX_ITERS=15`는 유지. 개선 후 테트리스 흐름은 6~9회면 충분하며, 한도 상향은 오히려 잘못된
전략을 연장할 뿐. 필요 시 기존 `UNITY_AGENT_MAX_ITERS` 환경변수로 오버라이드 가능.

### Step 4 — C#풍 float 인자 보정 (`mcp_client.py`)

`_coerce_args`는 이미 문자열 인자를 `json.loads`로 보정하지만 `0.9f`는 유효 JSON이 아니라
실패했다. 모듈 레벨 헬퍼로 최소 수정:

```python
_CS_FLOAT_SUFFIX = re.compile(r"(?<=[\d.])[fF]\b")

def _lenient_json(text: str):
    """C#풍 float 접미사('0.9f')를 허용해 JSON으로 파싱. 실패 시 예외 전파."""
    return json.loads(_CS_FLOAT_SUFFIX.sub("", text))
```

스키마가 array/number/bool로 선언된 인자에만 적용되므로 스크립트 내용·이름 등 문자열
파라미터는 절대 건드리지 않는다. 지난번 검증 에러로 반복 1회를 낭비했던 문제를 원천 차단.

### Step 5 — 반복 호출 루프 가드 (`agent.py` + `config.py`)

한 턴에서 같은 툴이 `LOOP_GUARD_THRESHOLD`(기본 4)회 호출되면, 기존 주입 스타일(대괄호
한국어 `user` 메시지)로 넛지를 **턴당 1회만** 삽입 + `on_warn`으로 CLI에도 표시:

```
[시스템] 같은 툴을 반복 호출하고 있습니다. 반복적인 오브젝트 생성은
unity_create_gameobjects 배치 툴 하나로 처리하거나, 큰 그리드는
스크립트의 Awake()/Start()에서 생성하세요.
```

`UNITY_AGENT_LOOP_GUARD=0`으로 끌 수 있다. Step 3 프롬프트가 처음부터 작동하면 이 층은
발동하지 않지만, 모델이 규칙을 놓치고 반복에 빠졌을 때의 실행 시점 안전망이다.

## 검증

라이브 Unity(6000.5.2f1, 브리지 포트 8723) + 실제 Ollama(qwen3-coder:30b)로 전 항목 통과.

| 항목 | 결과 |
|---|---|
| C# 재컴파일 | 콘솔 에러 0 (브리지로 refresh 트리거 후 확인) |
| 배치 스모크 (LLM 없이) | 4개 중 3개 생성 + 잘못된 primitive 1개를 index와 함께 에러 보고, 배치 내 부모 참조 동작, 빈 배열 거부 |
| 단일 생성 회귀 | `CreateOne` 리팩터 후 단일 경로 정상 + 부모 없을 때 고아 오브젝트 미생성 |
| `_lenient_json` 단위 | `[0.9f,0.9f,0.9f]`→`[0.9,0.9,0.9]`, `0.5F`→`0.5`, 문자열 내부 `f`는 보존, 쓰레기 입력은 여전히 예외 |
| MCP 툴 등록 | 총 19개 툴에 `unity_create_gameobjects` 포함, 스키마에 `$defs` 없음(평탄) |
| 루프 가드 단위 | 4번째 호출 직후 정확히 1회만 넛지 주입, `THRESHOLD=0`이면 미발동 |
| **실전 E2E** | "큐브 5개 일렬로" → 모델이 계층 확인 → 부모 생성 → **배치 1회**, 총 툴 호출 3회로 완료 |

E2E에서 모델은 프롬프트 규칙대로 개별 생성 없이 배치 툴을 스스로 선택했다.

## 반영 순서 (재현/롤아웃 시)

1. Step 1(C#) → Unity 에디터 포커스로 재컴파일 (도메인 리로드로 브리지 포트 호핑 가능하나
   기존 `Library/McpBridgePort.txt` + `_reconnect_if_port_changed`가 처리 — 추가 작업 없음).
2. Step 2·3은 **같이 배포** (프롬프트가 존재하지 않는 툴을 언급하면 환각 호출 유발).
3. Step 4·5는 독립적. 어느 경우든 실행 중인 `main.py`는 **재시작**해야 새 툴이 잡힌다
   (서버는 시작 시마다 새로 spawn).

## 알려진 한계

- 배치 툴은 3~50개 구간용. 그 이상은 프롬프트가 스크립트 경로로 유도하지만, 모델이 규칙을
  무시하고 대량 사양을 배치로 넘길 수는 있다 (500개 상한 + 토큰 압축 응답으로 완충).
- 배치 내 **중복 이름**은 `FindGameObject`가 첫 매치를 반환하므로 이후 조회/수정이 모호해진다
  — docstring이 "고유 이름 사용"을 지시하지만 강제하지는 않는다.
- 루프 가드는 "같은 툴 이름 N회"만 감지 — 서로 다른 툴을 번갈아 반복하는 패턴은 잡지 못한다
  (현재 실패 모드에는 충분).
