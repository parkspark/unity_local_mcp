# 버그 리포트: Unity MCP 브리지 포트 좀비

| 항목 | 내용 |
|---|---|
| 발생일 | 2026-07-03 |
| 심각도 | **높음** — 브리지 전면 불능, 에디터 재시작 없이는 복구 불가 (수정 전 기준) |
| 영향 범위 | `UnityMcpBridge.cs`를 쓰는 모든 클라이언트 (로컬 LLM 에이전트, Claude Code 동일) |
| 환경 | Windows 11, Unity 6000.5.2f1 (DX12), Mono 도메인 리로드 기본 설정 |
| 상태 | **해결** — 브리지 안정화 4건 + 호스트 대응 2건 |

---

## 1. 요약 (TL;DR)

Unity 에디터의 **도메인 리로드 중 일부 teardown 경로에서는 정리 이벤트(`beforeAssemblyReload`, `AppDomain.DomainUnload`)가 발화하지 않는다.** 이 경로를 타면 브리지의 TCP 리스닝 소켓을 닫아줄 주체가 사라져, 소켓이 **좀비 상태로 에디터 프로세스가 사는 동안 포트 8722를 영구 점유**한다. 새 도메인의 브리지는 바인드에 실패하고, 좀비는 커널 수준에서 연결을 받기만 하고 응답하지 않으므로 클라이언트는 타임아웃에 빠진다.

포트 하나를 두고 Unity 내부 동작과 싸우는 모든 시도(재시도, SO_REUSEADDR, 스레드 제거, 언로드 훅)가 실패하거나 부분 해결에 그쳤고, 최종 해결은 **싸움 자체를 회피하는 설계**였다: 포트를 뺏기면 다음 포트로 옮기고(8722→8741 호핑), 실제 포트를 파일로 알린다.

---

## 2. 증상

세 가지 겉모습으로 나타나 진단을 어렵게 했다:

1. **연결 거부** (`WinError 10061`) — 리스너 자체가 없음. 새 도메인이 바인드 실패 후 포기한 상태.
2. **수신 타임아웃** (`timed out`) — 리스너는 있는데 응답이 없음. 좀비 소켓이 커널 백로그로 연결만 받는 상태. `netstat`에 서버 쪽 `CLOSE_WAIT`가 쌓이는 것이 특징.
3. **Editor.log의 결정적 로그**:
   ```
   [McpBridge] Failed to start listener on port 8722:
   각 소켓 주소(프로토콜/네트워크 주소/포트)는 하나만 사용할 수 있습니다.
   ```

트리거는 **스크립트 컴파일을 동반한 도메인 리로드**(`unity_refresh_assets`, 새 `.cs` 임포트, 프로젝트 오픈)였다. 플레이 모드 전환 리로드에서는 대체로 재현되지 않아 초기에 혼선을 줬다 — 리로드 시점에 미처리 연결이 있느냐 없느냐의 차이였다.

## 3. 진단 타임라인 — 가설과 반증

실측 근거: `netstat -ano`, `C:\Users\park\My project\Logs\Editor.log`, 원시 TCP 프로브.

### 가설 1: 일시적 바인드 레이스 → 재시도 부재가 문제다
- **수정**: 바인드 실패 시 2초 간격 60초 재시도(`TryBindRetry`) + 에셋 임포트 워커 프로세스에서 브리지 기동 금지 가드.
- **결과**: 실패. 60초를 다 써도 포트가 풀리지 않음 → 점유가 일시적이지 않다는 뜻.

### 가설 2: 이전 도메인의 CLOSE_WAIT 연결이 주소를 점유한다 → SO_REUSEADDR
- **수정**: 리스너에 `SO_REUSEADDR` 설정.
- **결과**: **악화.** 두 가지 새 실패 모드 확인:
  - 옛 소켓(옵션 없는 세대)이 살아 있으면 `WSAEACCES`(액세스 거부)로 바인드 실패 — Windows에서 REUSEADDR 하이재킹은 **양쪽 소켓 모두** 옵션이 있어야 한다.
  - 양쪽 모두 옵션이 있으면 **같은 프로세스에 LISTENING 소켓 2개**가 공존(netstat로 확인)하고, 신규 연결이 죽은 쪽으로 라우팅되어 전부 타임아웃.

### 가설 3: accept 전용 스레드가 소켓을 고정한다
- **분석**: 네이티브 `accept()`에 블로킹된 스레드는 도메인 언로드 때 중단되지 못하고 살아남아, 자기가 든 리스닝 소켓 핸들을 프로세스 수명 내내 고정한다. (역설적으로 원래 설계에서는 이 스레드의 "언로드 시 중단 → 소켓 강제 닫힘"이 비공식 청소 경로로 작동해 대부분의 리로드를 버텨왔다.)
- **수정**: accept 스레드 제거, 메인 스레드 `EditorApplication.update`에서 논블로킹 `Pending()` 폴링으로 전환.
- **결과**: 부분 개선(스레드 고정 제거)이지만 **부팅 리로드에서 여전히 누수** — 소켓을 닫아줄 이벤트 자체가 안 오면 스레드 유무와 무관.

### 가설 4: `beforeAssemblyReload`가 부팅 리로드에 발화하지 않는다 → `DomainUnload` 훅 추가
- **수정**: `AppDomain.CurrentDomain.DomainUnload += StopListener`.
- **결과**: 실패. **이 teardown 경로에서는 둘 다 발화하지 않음**을 실측으로 확인. (프로젝트 오픈 부팅 리로드에서 재현율 100%)

### 가설 5: 틱 한 번 없이 버려지는 부팅 도메인에서 바인드하는 것 자체가 문제다
- **수정**: 생성자(cctor) 직접 바인드 금지, `EditorApplication.delayCall`로 지연 — 첫 에디터 틱까지 살아남은 도메인만 바인드.
- **결과**: **부팅 경로 해결** (부팅 로그에서 실패 없이 단일 `Listening` 확인). 그러나 세션 중 리로드에서 또 다른 누수 발생 — 정리 이벤트가 안 오는 teardown 경로가 부팅에만 있는 게 아니었다.

### 최종 결론: 포트와 싸우지 않는다
어떤 방어를 해도 "이벤트 없는 teardown"이 한 번 발생하면 그 포트는 프로세스 종료까지 회수 불가능하다. 따라서 **고정 포트를 전제하지 않는 설계**로 전환했다 (4절).

## 4. 근본 원인

```
[도메인 A] 리스닝 소켓 8722 보유
     │
     ▼  도메인 리로드 (일부 경로: beforeAssemblyReload ✗, DomainUnload ✗)
     │
     ├─ 소켓을 닫을 코드가 실행되지 않음
     │  → 소켓은 죽은 도메인의 관리 객체에 뿌리 박힌 채 GC/파이널라이즈도 안 됨
     │  → 커널은 계속 LISTENING: 연결 수락(백로그)만 하고 응답할 코드는 없음
     ▼
[도메인 B] cctor/delayCall에서 8722 바인드 시도
     → WSAEADDRINUSE, 재시도 무의미 (점유가 영구적)
```

핵심은 두 가지의 결합이다:

1. **Unity의 teardown 이벤트는 계약이 아니다.** 프로젝트 오픈 부팅 리로드를 포함한 일부 경로에서 `beforeAssemblyReload`와 `DomainUnload` 모두 실행되지 않는다.
2. **언로드된 도메인의 소켓은 회수되지 않는다.** 관리 스레드가 잡고 있으면(구 설계) 스레드 고정으로, 안 잡고 있어도(신 설계) 파이널라이즈가 오지 않아, 어느 쪽이든 핸들이 프로세스 수명 동안 산다.

부차 요인: 리로드 순간 미응답 상태로 버려진 클라이언트 연결(`CLOSE_WAIT`)도 로컬 주소를 점유해 재바인드를 방해한다. Unity가 백그라운드일 때 컴파일을 미루는 동작은 리로드 창을 길게 늘여 재현 확률을 높였다.

## 5. 최종 해결책

### 브리지 (`Assets/Editor/McpBridge/UnityMcpBridge.cs`) — 4건

| # | 수정 | 목적 |
|---|---|---|
| 1 | cctor 직접 바인드 금지, `EditorApplication.delayCall`로 지연 | 틱 없이 버려지는 부팅 도메인이 소켓을 만들지 않게 함 |
| 2 | accept 스레드 제거 → 메인 스레드 `Pending()` 폴링 | 네이티브 accept에 갇힌 스레드의 소켓 고정 원천 차단 |
| 3 | **포트 호핑**: 8722부터 20개 포트를 순차 시도, 성공한 포트를 `Library/McpBridgePort.txt`에 기록 | 좀비가 생겨도 즉시 우회 — 재시작 불필요 |
| 4 | `StopListener`가 활성 클라이언트 소켓 전부 close + 대기 큐 폐기, `beforeAssemblyReload`/`DomainUnload`/`quitting` 훅 유지, 60초 바인드 재시도 유지 | 이벤트가 오는 정상 경로에서는 깨끗하게 정리 (다층 방어) |

### 에이전트 호스트 (`unity_local_mcp`) — 2건

| # | 수정 | 목적 |
|---|---|---|
| 5 | `mcp_client.py`가 포트 파일을 읽어 `UNITY_MCP_PORT` 환경변수로 server.py에 전달 | **server.py 무수정**으로 호핑된 포트 추적 (env는 기존 지원 기능) |
| 6 | 도구 호출 실패 시 포트 파일 재확인 → 바뀌었으면 MCP 서버 서브프로세스 자동 재기동 (`_reconnect_if_port_changed`) | 세션 중 호핑에도 무중단 대응 |

**설계 원칙**: 좀비 발생을 100% 막으려 하지 않는다(막을 수 없음이 실증됨). 대신 발생해도 **아무도 눈치채지 못하게** 한다.

## 6. 검증

- Unity 재시작 후 부팅: `Listening on 127.0.0.1:8722` 단일 기록, 바인드 실패 없음 (수정 전에는 부팅마다 실패 로그).
- 좀비가 존재하는 라이브 세션에서 새 코드 컴파일 → 재시작 없이 `Listening on 127.0.0.1:8723`으로 자동 이동, 포트 파일 `8723` 기록 확인.
- 호스트가 포트 파일을 읽어 8723으로 `unity_ping`/`unity_read_console` 정상 응답.
- E2E: 계층 조회 → `PlayerMovement` 컴포넌트 부착 → 재검증까지 에이전트 무개입 완주.
- 미검증(코드만 완성): 세션 중 실시간 호핑 시 자동 재접속의 실전 발동. 다음 스크립트 컴파일 작업에서 자연 검증 예정.

## 7. 교훈

1. **플랫폼의 생명주기 이벤트를 계약으로 믿지 말 것.** "리로드 전에 항상 불린다"는 문서적 기대가 특정 경로에서 조용히 깨졌고, 이것이 모든 1차 수정의 실패 원인이었다.
2. **SO_REUSEADDR는 Windows에서 복구 수단이 아니다.** 하이재킹 조건(양쪽 옵션)과 이중 바인드 라우팅의 비결정성 때문에 증상을 바꿀 뿐 고치지 못하며, 이번 사례처럼 악화시킬 수 있다.
3. **에디터 확장에서 블로킹 네이티브 호출을 하는 전용 스레드는 부채다.** 도메인 리로드와 만나는 순간 자원 회수 불가 문제로 돌아온다. 메인 스레드 폴링이 몇 프레임의 지연을 감수할 가치가 있다.
4. **자원 소유권 싸움에서 못 이기면 자원을 바꿔라.** 다섯 번의 "올바른 정리" 시도보다 포트 호핑 한 번이 문제를 종결시켰다.
5. **증상 3종(거부/타임아웃/이중 리스너)을 구분해준 것은 `netstat`과 Editor.log였다.** 클라이언트 쪽 에러 메시지만으로는 전부 "브리지 다운"으로 보인다.

## 8. 남은 리스크

- 포트 호핑 범위는 20개(8722–8741). 한 에디터 세션에서 이벤트 없는 teardown이 20회를 넘으면 소진된다 — 현실적으로 도달하기 어렵고, 도달 시 에디터 재시작으로 전량 회수된다.
- `Library/McpBridgePort.txt`는 에디터 비정상 종료 시 마지막 포트로 남지만, 다음 부팅에서 8722 바인드 성공 시 즉시 갱신되므로 자가 치유된다.
- Claude Code 등 다른 MCP 클라이언트는 포트 파일을 모른 채 기본 8722로 접속한다 — 좀비가 없는 정상 상태에서는 문제없지만, 호핑이 일어난 세션에서는 `UNITY_MCP_PORT`를 수동 지정해야 한다.
