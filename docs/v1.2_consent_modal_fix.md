# v1.2 — "Script Updating Consent" 모달 자동 처리

v1.1 알려진 문제 #1 해결. 2026-07-04.

## 문제

로컬 모델이 Unity 6에서 개명된 구식 API(대표적으로 `Rigidbody.velocity`)로 스크립트를
작성하면, `unity_refresh_assets` 시 Unity의 API Updater가 **"Script Updating Consent"**
모달을 띄운다. 이 모달은 에디터 메인 스레드를 멈추므로 브리지도 무응답이 되고, 에이전트는
컴파일 대기 폴링을 타임아웃까지 반복하다 마비된다 (실측 5분+, 사람이 클릭해야만 해제됨).

GUI 모드에서 이 모달을 끄는 공식 방법은 없다 (`-accept-apiupdate`는 배치 모드 전용,
[Unity 포럼](https://discussions.unity.com/t/how-to-disable-script-updating-consent/918503)에서도 미해결).

## 해결: 3중 방어

| 층 | 파일 | 동작 | 검증 |
|---|---|---|---|
| ① 예방 | `agent.py` | 시스템 프롬프트에 Unity 6 구식 API 금지 규칙 (`rb.linearVelocity`, `FindFirstObjectByType`) | 텍스트 규칙 |
| ② 차단 | `local_tools.py` | `unity_write_script`가 쓰기 전에 `.velocity` → `.linearVelocity` 자동 교정, 결과에 `auto_fixed` 표기 | ✅ 디스크 파일 교정 + `auto_fixed` 반환 확인 |
| ③ 해제 | `winmodal.py` + `mcp_client.py` | 모달이 떠도 Win32 API로 감지해 "전체 동의" 버튼 클릭 | ✅ 네이티브 테스트 창으로 감지→클릭 확인 |

**설계 원칙**: ①은 발생 빈도를 줄이고, ②는 가장 흔한 원인을 원천 차단하며, ③은 무엇이 새어
나와도 막는 최종 안전망. 세 층이 독립적이라 하나가 실패해도 나머지가 받친다.

### ③ winmodal.py 동작

- `FindWindowW(None, "Script Updating Consent")`로 창 탐지
- `EnumChildWindows` + 정확한 텍스트 매칭으로 **"Yes, for these and other files that
  might be found later"** 버튼만 클릭 (다른 버튼/창은 절대 건드리지 않음)
- `SendMessageTimeout(BM_CLICK)` — 포커스를 뺏지 않고, 멈춘 대화상자 스레드에 호스트가
  함께 갇히지 않도록 타임아웃 클릭
- 창/버튼을 못 찾으면 아무것도 안 하고 `False` → 기존 수동 처리로 자연 강등 (안전 기본값)

호출 지점 (`mcp_client.py`):
- `_wait_for_compile()` 폴링 루프 각 반복 (모달은 refresh 직후 등장)
- `call()`의 재시도 분기 (`Cannot reach` / `Empty response` — 모달 마비도 이 경로)

## 설정

`UNITY_AGENT_AUTO_CONSENT=0`으로 ③층을 끌 수 있다 (기본 켬).

## 추가 (같은 메커니즘으로 확장): 씬 수정 모달의 Ignore 자동 클릭

플랫포머 재검증 중, consent 모달은 안 떴지만 **"The open scene(s) have been modified
externally"** 모달이 dirty 씬에서 브리지를 막는 것이 확인됨 (SceneAutoReload는 작업 보호를
위해 dirty 씬에 의도적으로 개입하지 않음). `winmodal.dismiss_scene_modified_ignore()`를
추가해 **Ignore**(현재 에디터 상태 유지 — 데이터 손실 없는 선택)를 자동 클릭하도록 확장.
실제로 떠 있던 라이브 모달을 대상으로 클릭 → 브리지 즉시 복구를 실측 검증함.

## 최종 검증 (플랫포머 E2E 재실행)

- 모달 등장 0건, 브리지 끊김("Cannot reach") 0건, 컴파일 대기 정상 완료 2회
- `rb.velocity` 사용 0회 — ①층(프롬프트 규칙)이 소스 단계에서 작동, 모델이 처음부터
  `linearVelocity` 사용
- 플레이 모드 왕복 2회, 콘솔 에러 0, 씬에 9개 오브젝트(바닥·플랫폼 3·플레이어·장애물·목표)
  완성 — 게임 뷰 스크린샷으로 확인

## 알려진 한계

- 모달 창 제목/버튼 텍스트는 Unity 버전 의존적. 다르면 매칭 실패 → 수동 처리로 강등
  (기존과 동일, 악화 없음).
- consent 모달의 ③층은 동일 제목·버튼의 네이티브 테스트 창으로 검증됨 (실 모달은 ①·②층이
  원인을 제거해 재현되지 않음 — 그것이 목표였음). 씬 수정 모달 쪽은 실 모달로 검증 완료.
