## 3. 발견된 문제

### P0 — 다음 사이클 전에 우선 수정

#### O-001 요청 내부의 씬 경로 모순을 실행 전에 차단하지 않음

- 제작 대상은 `Platformer25D_MVP_22.unity`였지만 합격 조건은
  `Platformer25D_MVP.unity`를 요구했다.
- `VerificationSpec.asset_paths`가 두 경로를 모두 필수 자산으로 수집해 성공이
  구조적으로 불가능했다.
- 개선:
  - 실행 전 요구사항 정규화 단계에서 서로 다른 단일 씬 경로를 발견하면 제작을 시작하지
    말고 충돌을 명시한다.
  - 사용자가 명시한 “최종 합격 조건”을 우선할지, 첫 산출물 경로를 우선할지 정책을
    코드로 고정한다.
- 회귀 테스트: 서로 다른 두 `.unity` 경로가 단일 씬 요청에 들어오면 도구 호출 0회로
  `conflicting_scene_paths`를 반환해야 한다.

#### O-002 dirty Untitled 씬 사전 점검 누락과 네이티브 Save Scene 모달

- 첫 `unity_create_scene`은 `active scene has unsaved changes`로 실패했다.
- 이어진 `unity_save_scene {}`는 경로가 없는 Untitled 씬에서 네이티브 저장 창을 열고
  MCP 호출을 타임아웃시켰다.
- 개선:
  - 제작 전 `unity_get_state`로 `path`, `isDirty`를 확인하는 preflight를 강제한다.
  - `unity_save_scene(path=...)`를 지원하거나 `unity_create_scene`에 안전한
    `save_current_as`/`discard_current` 정책을 명시적으로 제공한다.
  - 네이티브 파일 대화상자를 열 수 있는 무인 도구 호출은 사전 차단한다.
- 회귀 테스트: dirty Untitled에서 새 씬 요청을 실행해도 OS 대화상자가 뜨지 않고 지정한
  경로로 진행돼야 한다.

#### O-003 도메인 리로드 후 브릿지 재기동이 Unity 포커스에 의존

- 세 스크립트의 실제 컴파일은 각각 약 3.10초, 1.33초, 1.35초에 성공했다.
- 도메인 리로드 뒤 `[McpBridge] Listening`이 발생하지 않아 첫 컴파일에서는 호스트가
  잘못된 “90초가 지나도 아직 컴파일 중” 메시지를 반환했다.
- Unity 창을 활성화한 뒤에야 브릿지와 `unity_refresh_assets`가 복구됐다.
- 개선:
  - `winfocus.focus_unity()`의 성공 여부를 검증하고 실패 시 제한적으로 재시도한다.
  - “컴파일 중”과 “컴파일 완료 후 브릿지 미기동”을 다른 상태로 기록한다.
  - 브릿지 정적 초기화에서 `delayCall` 등록 후 player-loop 갱신을 요청하거나, 안전한
    세션 표식으로 부트 도메인과 정상 리로드 도메인을 구분해 즉시 바인딩한다.
  - `_wait_for_compile`에 브릿지 독립적인 컴파일 완료 증거(Editor.log/build artifact)와
    브릿지 liveness 단계를 분리한다.
- 회귀 테스트: Unity가 백그라운드인 상태에서 스크립트를 3회 연속 수정해도 각 호출이
  30초 안에 복구되고 사람 포커스 개입이 0회여야 한다.

#### O-004 제작 단계 실패 후 검증 복구기가 전체 제작을 대신함

- 플래너는 마일스톤 1 실패 뒤 나머지 4개 마일스톤을 미착수로 중단했다.
- 그 뒤 독립 검증기가 13개 실패를 수집하고 repair 모델이 Player, 스크립트, 카메라,
  레벨까지 사실상 처음부터 만들었다.
- builder와 repair의 책임 경계가 무너져 planner ledger, 산출물 범위, 금지된 단계가
  repair 컨텍스트에 보존되지 않았다.
- 개선:
  - `build_stage_success=false`이고 최소 제작 계약(Player/필수 스크립트/씬)이 없으면
    검증 repair로 넘어가지 말고 실패한 제작 마일스톤을 재계획한다.
  - repair에는 “기존 성공 산출물 + 실패 항목 + 금지/제거된 계획 단계”를 구조화해 전달한다.
- 회귀 테스트: 마일스톤 1이 실패하면 repair가 `LevelLoader` 같은 후속 산출물을 만들지
  않아야 한다.

- Player 부재가 확정된 최초 검증에서도 같은 GameObject 조회, Play 진입, 키 입력,

- 개선:
  - 자연어 숫자/방향/시간을 `VerificationSpec`의 명시 필드로 파싱한다.
  - 각 합격 조건에 `measured/pass/fail/blocked` 상태와 원시 측정값을 영수증에 남긴다.
  - 측정되지 않은 조건은 자동으로 실패시킨다.

#### O-007 수정 사이클이 실패를 줄이지 못해도 끝까지 반복

- 실패 수가 `13 → 7 → 8 → 9`로 후반 두 사이클에서 악화됐다.
- 현재 무진전 감지는 동일한 실패 집합 반복에 치우쳐 실패 집합이 조금 바뀌면 회귀를
  놓친다.
- 개선:
  - 실패 개수뿐 아니라 새로 생긴 실패, 해결된 실패, Jaccard 유사도, 치명도 가중치를
    비교한다.
  - 한 사이클에서 새 P0 실패가 생기거나 총점이 악화되면 즉시 중단/롤백 후보로 표시한다.
  - repair 프롬프트에는 직전 변경과 새 회귀를 명시한다.

#### O-008 계획에서 제거한 범위가 repair에서 되살아남

- 플래너는 “요청에 없는 데이터 주도 레벨 단계”를 제거했다.
- 세 번째 repair는 `level1.json`을 조회하고 `LevelLoader.cs`, `LevelLoader` 오브젝트,
  레벨 JSON을 새로 만들었다.
- 이 변경 때문에 최종 씬이 dirty가 되어 `scene_not_saved`가 다시 발생했다.
- 개선:
  - 계획 정규화 결과에 `forbidden_scope`를 추가해 모든 builder/repair 프롬프트와
    도구 정책에 전달한다.
  - 마지막 mutation 뒤에는 반드시 저장/clean 확인을 수행하는 커밋 배리어를 둔다.

#### O-009 중복 이름과 잘못된 오브젝트 타입을 허용

- 개선:

  - 필수 오브젝트 수정에는 create보다 get/ensure를 우선하는 전용 도구를 제공한다.

#### O-010 반복 조회와 iteration 예산 낭비

  - 반복 경고만 출력하지 말고 일정 횟수 뒤 배치 호출로 host-side 전환한다.

  - 검증기는 최종 bitmask가 요구 비트를 모두 포함하는지 측정한다.

## 4. 권장 개선 순서

한 번에 전부 고치면 어느 변경이 성공률을 올렸는지 알 수 없다. 아래 묶음별로 같은
프롬프트와 같은 깨끗한 Unity 기준 상태를 사용해 A/B 비교한다.

### Cycle A — 실행 가능성

1. 씬 경로 모순 preflight
2. dirty Untitled 처리와 네이티브 Save Scene 방지
3. 도메인 리로드 후 브릿지 liveness/포커스 복구

합격 기준: 사람 개입 0회, 연결 타임아웃 0회, 초기 제작 마일스톤 진입 성공.

### Cycle B — 검증 비용과 정확도

1. prerequisite DAG와 치명 실패 조기 단락
2. 측정 전 Player 상태 리셋
3. 15개 합격 조건의 1:1 evidence schema
4. 중복 GameObject snapshot 제거

합격 기준: 실패 프로젝트에서 불필요한 키 입력 0회,
`unity_get_gameobject` 40회 이하, 모든 조건이 `pass/fail/blocked` 중 하나를 가짐.

### Cycle C — repair 통제

1. builder 실패와 repair 분리
2. `forbidden_scope` 전달
3. 중복 이름 방지
4. 실패 회귀 감지와 마지막 mutation 후 save barrier

합격 기준: 실패 수가 사이클마다 단조 감소하고, 요청 밖 LevelLoader 생성 0건,
최종 `isDirty=false`.

### Cycle D — 모델 산출물 품질

1. 요청 기반 C# 정적 검사
2. Rigidbody enum bitmask 안전 설정
3. 배치 생성 도구 우선 정책

합격 기준: Play 전에 known 정적 위반 0건, 5개 이상 플랫폼을 한 번의 batch 호출로 생성.

## 5. 다음 실행에서 반드시 수집할 비교 지표

매 사이클에 아래 값을 같은 형식으로 남긴다.

- 성공/실패와 최종 failure IDs
- 총 시간, 첫 유효 mutation까지 시간
- 모델 응답/도구 호출/도구 오류 수
- 도구별 호출 수와 중복 조회 비율
- 컴파일 횟수, 실제 컴파일 시간, 브릿지 복구 시간
- native modal 수, 사람 개입 수
- 각 repair 전후 `resolved/new/persisted` 실패 집합
- 요청 밖 생성 자산 수, 중복 이름 수, 마지막 씬 dirty 여부
- 15개 합격 조건 각각의 `pass/fail/blocked/not_measured`

개선 성공은 단순히 최종 성공 여부만이 아니라 **사람 개입 0, 실패 단조 감소, 측정 누락 0,
도구 호출 감소**를 함께 만족할 때로 정의한다.
