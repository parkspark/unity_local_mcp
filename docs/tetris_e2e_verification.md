# 테트리스 E2E 통합 검증 및 제작 기록

2026-07-06~07. "unity_local_mcp로 빈 씬에서 테트리스를 만들어 플레이 모드로 플레이 가능한가"를
실제로 시켜 보며 v1.3/v1.4/v1.4.1 개선을 통합 검증한 기록.

## 목표

로컬 에이전트(qwen3-coder:30b)에게 명령만 주고, 실제 제작(스크립트 작성·컴파일·씬 배선·플레이
검증)은 전부 에이전트가 MCP 툴로 수행하게 해서 **플레이 가능한 테트리스**를 완성 — 그 과정에서
최근 개선들이 실전에서 작동하는지 확인.

## 1. 파이프라인 검증 — 개선 사항 전부 실전 발동

여러 차례 에이전트 런에서 다음이 확인됨:

| 개선 | 실전 확인 |
|---|---|
| v1.3 배치 툴 | `unity_create_gameobjects`로 GameBoard+Player 한 번에 생성 |
| v1.3 스크립트-그리드 규칙 | 10×20 셀을 개별 200회 호출 대신 스크립트 런타임 생성으로 전환 |
| v1.3 tool-call 누수 salvage | `<function=…>` 텍스트 누수를 자동 복구·실행(혼합 응답 병합 포함) |
| v1.3 루프 가드 | 같은 툴 반복 시 넛지 발동 |
| v1.4 스크립트 삭제 | **에이전트가 스스로** 충돌 스크립트를 `unity_delete_script`로 제거해 컴파일 복구 |
| v1.4.1 stale 콘솔 | 컴파일·런타임 에러를 매번 진실하게 반영, 유령 파일 무한 추격 재발 0 |

이전에 물어봤던 "너무 많은 툴 호출로 중단"·"삭제 안 됨"·"유령 파일" 문제는 재현되지 않았다.

## 2. 도중 문제: 브리지 프리즈 — 범인은 "scene modified" 모달

첫 자율 빌드가 `TetrisBoard.cs` 컴파일 중 브리지 "Cannot reach"로 죽어 끝까지 안 돌아왔다.
조사 결과:
- Unity 프로세스는 **정상 응답**(IsHungAppWindow=False), 브리지 리스너만 완전 다운(어느 포트에도 없음).
- 원인은 **"The open scene(s) have been modified externally"** 모달(버튼 Reload/Ignore)이 에디터
  메인 스레드를 막은 것 — Unity 메인 창이 `en=False`로 비활성.
- `winmodal.dismiss_scene_modified_ignore()`로 **Ignore**(현재 상태 유지, 데이터 손실 없음) 클릭 →
  에디터 즉시 풀림 → 브리지 8722로 재바인딩.

**교훈**: 자율(사람 없이) 실행에서는 모달이 에디터+브리지를 얼린다. `mcp_client`의 모달 처리는
compile-wait/재시도 시점에만 도는데, 그 창 밖에서 뜨면 놓친다. → **2초 폴링 모달-워처 스레드**를
붙여 재실행하니 무중단으로 완주(자세한 후속 권장은 아래). 상세: [[autonomous-modal-freeze]].

## 3. 테트리스 제작 — 에이전트 주도 + QA 루프

방침: 인스펙터 배선이 필요 없는 **자기완결형 단일 스크립트**로 유도(그래야 반복 한도 안에 완주).
사람은 명령 + QA(실제 5초+ 플레이 후 콘솔/스크린샷 확인) 피드백만.

| 단계 | 에이전트 동작 / QA 결과 |
|---|---|
| 1차 빌드 | 자기완결형 `TetrisGame.cs` 작성(누수됐지만 salvage 복구)·컴파일 0에러·부착·플레이. 옛 `TetrisController` 충돌 예외 발견 → **`unity_delete_script`로 충돌 스크립트 정리** |
| QA① | 6초 플레이 → `Tag: PlacedPiece is not defined` 예외(정의 안 된 태그를 착지 시 사용). "에이전트의 이른 확인"이 놓친 버그를 긴 플레이로 포착 |
| 2차 수정 | 태그 제거 → 배열 기반 점유 관리로 수정 → 런타임 에러 0 |
| QA② (시각) | 화면이 회색 벽뿐(보드 전체를 회색 큐브로 채움 + 카메라 미조정) → 요구사항 피드백 |
| 3차 수정 | 배경 큐브 제거 → **청록 I자 조각 낙하 + 정착 블록이 화면에 실제 렌더링** |

## 4. 최종 결과

- `Assets/Scripts/TetrisGame.cs` (자기완결형 단일 스크립트) — `TetrisManager`에 부착, 프로젝트에
  스크립트는 이 하나만 남음(에이전트가 잔재 정리).
- 플레이 모드에서 **조각 스폰 → 낙하 → 정착**이 실제 동작, **컴파일/런타임 에러 0**(6초 플레이 검증).
- Play만 누르면 시작(인스펙터 배선 불필요). 조작: 방향키 이동/드롭, 스페이스·방향키 회전/하드드롭.

## 5. 솔직한 한계

- **카메라 프레이밍**: 아직 perspective라 조각이 작고 높게 보인다. orthographic 정렬은 30B 로컬
  모델이 여러 번 시도에도 깔끔히 못 맞췄다.
- **게임 로직 폴리시**: 에러 없이 돌지만, 정착 블록 색 유지 등 세부는 완벽하지 않다. "에러 0"이
  "시각적으로 완성된 게임"을 보장하지 않으며, 스크린샷 기반 QA(사람/비전)가 있어야 잡힌다.
- 즉 **파이프라인·툴·개선은 견고하게 작동하지만, 최종 산출물 품질은 로컬 모델 역량에 좌우**된다.

## 6. 후속 권장

- **상시 모달-워처를 실제 에이전트(main.py)에 내장** 검토 — 현재는 compile-wait/재시도 시점에만
  모달을 처리한다. 2초 폴링 데몬을 붙이면 자율 실행이 크게 견고해진다(이번 테스트로 효과 확인).
- 시각 검증을 워크플로에 포함 — `unity_screenshot` + (선택)비전 모델로 "에러 0"을 넘어 "보이는
  결과"까지 자동 확인. [v1.1 #4](v1.1_known_issues.md)의 구조적 한계와 같은 맥락.
