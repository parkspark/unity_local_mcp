# v1.4 — 스크립트 파일 직접 삭제 + 혼합 응답 tool-call 복구

2026-07-06. 에이전트가 기존 스크립트 파일을 삭제하지 못해 컴파일 실패에서 못 빠져나오던 문제 해결.

## 문제

빈 씬에서 테트리스를 만들다, 이전 세션이 남긴 `Assets/Scripts/TetrominoBlock.cs`가 중복 클래스
(CS0111)로 컴파일을 깼다. 에이전트는 이 파일을 **삭제할 수단이 없어** 우회(다른 스크립트 수정)만
반복하다 막혔다. 실제 로그: `"Unity Editor 도구로는 직접 파일을 삭제할 수 없습니다"`.

로컬 파일 툴은 `unity_write_script`·`unity_read_script`뿐이었고 삭제가 없었다.
`unity_delete_gameobject`는 씬 오브젝트 전용이라 디스크의 `.cs`에는 무력하다.

## 해결 1: `unity_delete_script` 로컬 툴 ([local_tools.py](../local_tools.py))

브리지 C# 커맨드가 아니라 **호스트(Python)가 디스크에서 직접 삭제**하는 로컬 툴로 구현.
write/read와 동일한 패턴이며, 삭제 후 `unity_refresh_assets`로 Unity가 반영한다.

**왜 브리지가 아니라 로컬인가** — 이 기능이 필요한 순간은 정확히 *프로젝트 컴파일이 깨진 상태*다.
이때 브리지에 새 커맨드를 넣으려면 브리지(`Assets/Editor/...`)를 재컴파일해야 하는데, 깨진
스크립트 때문에 프로젝트 전체가 컴파일 불가 → **닭-달걀**. 호스트 측 파일 삭제는 Unity 컴파일
상태와 무관하게 항상 동작한다.

동작:
- 기존 `_resolve()` 경로 가드 재사용 — `Assets/` 아래 `.cs`만 허용, `..` 탈출 차단.
- **브리지 자기보호**: `Assets/Editor/McpBridge/` 아래 경로는 거부 (에이전트가 자기 제어 채널인
  `UnityMcpBridge.cs`를 지워 브리지를 죽이는 사고 방지).
- `.cs` + 동반 `.cs.meta`를 함께 삭제 (meta를 남기면 Unity가 경고). 반환에 `deleted`,
  `meta_deleted`, `next_step`(refresh 안내) 포함.
- 파일 없으면 `file not found` 에러.

프롬프트 규칙 1줄 추가([agent.py](../agent.py)): 중복/충돌 스크립트는 그대로 두지 말고
`unity_delete_script`로 삭제 후 `unity_refresh_assets`.

## 해결 2 (필수): 혼합 응답 tool-call 복구 ([agent.py](../agent.py))

**툴만 추가해선 작동하지 않았다.** 라이브 E2E에서 모델이 `unity_delete_script`를 호출하려 했으나
**실행되지 않고 조용히 사라졌다.** 원인은 [[qwen3-coder-toolcall-leak]] 복구 로직의 갭:

qwen3-coder는 한 응답에서 **일부는 정상 tool-call, 일부는 `<function=...>` 텍스트로 흘리는**
'혼합 응답'을 낸다. 실제로 관찰된 응답:

```
<function=unity_delete_script><parameter=path>Assets/Scripts/DeleteMe.cs</parameter></function>  ← 텍스트로 샘
+ unity_refresh_assets  ← 정상 tool-call
```

기존 salvage 로직은 `if not calls`(정상 호출이 0건일 때)에만 작동해서, 정상 refresh가 있으면
**누수된 delete를 통째로 버렸다.** 모델은 삭제 성공으로 착각(실제론 파일 잔존)하고 원래 실패
패턴으로 되돌아갔다.

수정 — `_merge_leaked_calls()`로 정상 호출과 누수분을 **병합**:
- 누수분을 **앞쪽**에 배치. 모델은 보통 주 동작(삭제)을 먼저 서술하고 후속(refresh)을 정상 호출로
  내므로, 이 순서가 의도와 일치한다 (delete → refresh).
- 이름+인자(`json.dumps` 정렬)로 **중복 제거** — 같은 호출이 텍스트·정상 양쪽에 있어도 1회만 실행.
- 누수 없으면 무변경, 순수 누수(정상 0건)면 기존처럼 전부 복구.

이 수정은 delete뿐 아니라 배치 생성 등 **모든 툴의 혼합-응답 신뢰성**을 함께 높인다.

## 검증

| 항목 | 결과 |
|---|---|
| `unity_delete_script` 단위 (Unity 불필요) | .cs+.meta 삭제, meta 없을 때 처리, not found, 브리지 자기보호(백슬래시 포함), `_resolve` 가드 회귀 — 6/6 |
| `_merge_leaked_calls` 단위 | 혼합(누수 delete 먼저 + 정상 refresh), 중복 제거, 순수 누수, 무누수, 다중 누수 — 5/5 |
| **라이브 E2E (수정 전)** | 누수된 delete가 버려져 파일 잔존, 모델은 삭제 성공 착각 → **실패** |
| **라이브 E2E (수정 후)** | 누수 delete 복구·실행(순서: delete→refresh), 더미 디스크에서 제거, `unity_delete_script` 1회/총 3회 → **통과** |

수정 후 모델은 남은 `TetrominoBlock.cs` 에러도 "DeleteMe와 무관한 기존 문제"로 정확히 구분했다.

## 수정 파일

| 파일 | 변경 |
|---|---|
| [local_tools.py](../local_tools.py) | `unity_delete_script` (NAMES+SCHEMAS+call, `_resolve` 재사용, 브리지 자기보호) |
| [agent.py](../agent.py) | `_merge_leaked_calls`(혼합 응답 복구) + 프롬프트 삭제 규칙 1줄 |

mcp_client.py·config.py·server.py·UnityMcpBridge.cs **수정 없음.** `main.py`는 재시작해야 새 툴 반영.

## 알려진 한계 / 후속

- 혼합 응답에서 정상·누수 호출의 **진짜 실행 순서**는 채널이 달라 복원 불가 — "누수분 먼저"는
  관찰된 패턴에 맞춘 휴리스틱이다. 대부분의 (동작→refresh) 흐름엔 맞다.
- 사용자 프로젝트의 `Assets/Scripts/`에는 실패한 세션들이 남긴 **중복 테트리스 스크립트**
  (`TetrominoBlock.cs`, `Tetromino.cs`, `TetrisGame*.cs` 등)가 여럿 있어 컴파일이 깨진 상태다.
  이제 에이전트가 `unity_delete_script`로 정리할 수 있지만, 어떤 파일을 남길지는 사용자 판단이
  필요하다.
