# v1.7 — 호스트 측 플래너 · 데이터 주도 레벨 · 키 입력 시뮬레이션

## 요약

v1.6까지의 두 가지 근본 한계를 해결한다:

1. **포괄적 명령 불가** — 플래너 없는 단일 ReAct 루프(MAX_ITERS=15)라 "레벨 3개짜리
   플랫포머 만들어줘" 같은 요청을 모델이 한 번에 풀어야 했다.
2. **게임플레이 검증 불가** — 레벨 데이터 개념과 키 입력 시뮬레이션이 없어
   "실제로 플레이 가능한가"를 자동 확인할 수 없었다 (v1.6 리포트의 미검증 항목).

설계 철학은 유지: **모델은 제안자, 호스트는 결정자.** 모든 신규 경로는 실패 시
기존 단일 루프로 우아하게 강등된다.

## 축 1 — 호스트 측 플래너 (`planner.py`)

```
run_turn ──(소규모)──→ _run_single (기존 v1.6 루프, 동작 무변경)
        └─(대규모)──→ make_plan (도구 없는 별도 호출, Ollama format=스키마로 JSON 강제)
                       → _run_plan: 마일스톤별
                          [fresh history + TaskContract.for_milestone + _react_loop]
                          → ArtifactLedger 갱신 → 다음 마일스톤 프롬프트에 주입
```

- **대규모 판별**: `looks_large()` 결정적 휴리스틱(레벨×숫자, 장르+생성동사, 200자+생성동사).
  참일 때만 플래닝 호출 1회. 오판 비용: false negative는 기존 동작, false positive는
  `mode=single` 강등. `UNITY_AGENT_PLANNER=auto|always|off`.
- **계획 JSON 3중 방어**: ① Ollama structured output(`format=<json schema>`)으로 디코딩
  수준 제약 ② 코드펜스 제거 + 중괄호 추출 + `_lenient_json` salvage ③ temperature 0
  재시도 → 전부 실패 시 단일 루프 폴백.
- **호스트 정규화** (`validate_plan`): id 부여, 경로 `\→/` + `Assets/` 강제(위반 드롭),
  마일스톤 2~6개 클램프, goal 700자 캡.
- **마일스톤 실행**: 마일스톤마다 fresh 히스토리(32K 컨텍스트 보호) + 자체 TaskContract
  + 시작 시 `unity_ping`(도메인 리로드/포트 호핑 흡수). 완료 판정 3중:
  ① 모델의 무툴 종료 ② `missing_verification()==[]` ③ **deliverables 파일 존재 확인**
  (`os.path.exists`, 호스트 결정적).
- **마일스톤 간 상태 전달**: 히스토리 이월 없음. `ArtifactLedger`가 성공한 도구 결과에서만
  산출물 경로(scripts/levels/scenes/objects/screenshots, 카테고리별 10개 캡)를 수집해
  다음 마일스톤 프롬프트에 주입.
- **실패 처리**: 마일스톤당 재시도 1회(실패 원인 1줄 주입), 최종 실패 시 계획 중단 후
  `ledger.report()`("m1 ✓ / m2 ✗ (원인) / 산출물 목록")를 사용자에게 보고.
  전체 안전판 `PLAN_MAX_TOTAL_ITERS=60`.

## 축 2 — 데이터 주도 레벨

- **저장 위치**: `Assets/StreamingAssets/Levels/*.json`. JSON은 컴파일을 유발하지 않아
  **레벨 수정 → 즉시 플레이 검증**이 가능하다 (refresh 90초 폴링 생략).
  한계: Android/WebGL 빌드에서는 StreamingAssets 직접 File IO 불가 — 에디터 검증
  범위에서는 문제없음.
- **스키마** (`level_schema.py`, jsonschema): `{version:1, name, next_level(파일명|null),
  player_spawn:[x,y,z], goal:{position,size?}, objects[1..200]:{type∈platform|hazard|
  decoration, name?, position, size?, color?}}` + 의미 검사(platform≥1, goal≠spawn).
  오류는 사람이 읽는 문장 ≤5건으로 반환되어 그대로 모델 피드백이 된다.
- **canonical 로더** (`templates/LevelLoader.cs`): 모델이 로더를 작성하지 않는다 —
  `unity_install_level_loader`가 호스트 템플릿을 `Assets/Scripts/LevelLoader.cs`로 설치.
  근거: ① Python 스키마 ↔ C# JsonUtility 필드명 정합성은 호스트 결정 영역
  (`test_level_schema.py`가 정적으로 잠금) ② 모델 예산을 창작 영역(PlayerController,
  레벨 데이터)에 집중 ③ 결정적 콘솔 마커가 기계 검증 증거:
  - `[LevelLoader] Loaded <name>: N platforms, ...`
  - `[LevelLoader] GOAL reached: <name>` → next_level 체인 로드
  - `[LevelLoader] ALL LEVELS CLEAR`
- **TaskContract 통합**: 레벨 경로 게이트, `unity_write_level` 성공 시 런타임 검증
  (play→wait→console) 강제, 로더 설치는 스크립트 쓰기와 동일한 컴파일 사이클 요구.

## 축 3 — 키 입력 시뮬레이션 (`unity_send_key`)

브리지(`UnityMcpBridge.cs`)에 `send_key` 커맨드 추가 (v1.7부터 브리지/서버 수정 허용).

- **메커니즘**: `InputSystem.QueueStateEvent(Keyboard.current, new KeyboardState(...))` —
  에디터 메인스레드에서 큐잉하면 다음 프레임의 게임 코드 `Keyboard.current.*`에 반영.
- **KeyboardState는 전체 스냅샷**이므로 브리지가 `heldKeys` 집합을 유지하고 항상
  합집합을 큐잉 (A 홀드 중 B 탭이 A를 릴리스하지 않도록).
- **인터페이스**: `unity_send_key(key, action=tap|press|release, duration=0.15)`.
  tap은 즉시 응답하고 릴리스는 `ProcessQueue`(에디터 update 루프)에서 비동기 소화 —
  단일 요청-응답 프로토콜 유지. 플레이 종료/도메인 리로드 시 상태 클리어.
  콤보: `press right → tap space → release right`.
- **정책**: 플레이 모드 밖 호출은 TaskContract가 차단. 요청에 입력 키워드가 있거나
  마일스톤 verify에 `input`이 있으면 완료 전 입력 시뮬레이션을 강제.

### 실측: Unity 백그라운드 한계 (중요)

| 상태 | send_key | 비고 |
|---|---|---|
| Unity 포커스 (Game View) | ✅ 동작 | 브리지가 `GameView.Focus()`로 에디터 내부 포커스 확보 |
| Unity 최소화 | ❌ **플레이어 루프 전체 정지** | 입력만이 아니라 자동 낙하 등 게임 자체가 멈춤 (실측) |

→ 호스트가 `unity_send_key` 직전 `winfocus.focus_unity()`를 기본 실행
(`UNITY_AGENT_FOCUS_INPUT=1`). 컴파일 대기의 `FOCUS_UNITY_ON_COMPILE`과 같은 패턴.

검증 절차(시스템 프롬프트로 유도): play → `unity_get_gameobject Player`(위치1) →
`press` → wait → `release` → `unity_get_gameobject Player`(위치2) → 좌표 비교.
판단은 모델(제안자) 몫이되 증거(두 좌표)가 트랜스크립트에 남는다.

### 브리지 단독 검증 (Phase A)

기존 `LocalMcpTetrisV2` 씬에서 leftArrow tap 3회 → 피스 X좌표 -0.5 → -1.5 → -2.5
(탭당 정확히 1칸 이동), 런타임 에러 0. 컴파일 에러 0.

### 레벨 시스템 단독 검증 (Phase B)

LLM 없이 결정적 절차로: 로더 설치 → 컴파일 클린 → `LevelSystemTest.unity` 씬 구성
(LevelManager+LevelLoader, Player Cube+Rigidbody) → `level1.json` 작성(platform 1,
hazard 1) → 플레이:

```
[LevelLoader] Loaded Test Level 1: 1 platforms, 1 hazards, 0 decorations, goal at (6.00, 1.00, 0.00)
LevelRoot children: ['Ground', 'hazard', 'Goal']
```

스키마 검증 거부도 확인: platform 없는 레벨 → `level validation failed: objects must
contain at least one platform ...` (파일 미작성).

실행 중 통합 테스트에서 발견해 수정한 호스트 결함:

- `unity_refresh_assets` 결과 JSON 뒤의 compile-wait 안내 때문에 성공 판정이 실패하던 문제
  → 선두 JSON만 `raw_decode`하여 상태 판정.
- 전체 계획 예산을 실제 사용량이 아닌 마일스톤 최대치로 선차감하던 문제
  → `_react_loop`의 실제 iteration 수만 차감.
- scene save보다 play를 먼저 안내해 플레이 중 저장을 시도하던 문제
  → edit-mode 검증/저장을 play 요구보다 먼저 제시.
- 입력 도구 호출과 전후 조회만으로 이동 성공 처리하던 문제
  → 호스트가 Player 좌표를 파싱해 실제 변화(`>0.001`)를 결정적으로 요구.
- 모델이 3D 레벨에 `Rigidbody2D` 이동 스크립트를 생성한 문제
  → 시스템/플래너 프롬프트에 3D Rigidbody 제약 추가, E2E 스크립트 수정 후 재검증.

## 남은 권고

1. **골 도달 E2E의 완전 자동화**: 현재 이동 검증(좌표 비교)까지 자동. "레벨 1 →
   GOAL reached → 레벨 2 로드"의 연쇄를 입력 시뮬레이션만으로 통과하려면 모델이
   점프 타이밍을 맞춰야 해 아직 불안정 — 골 위치를 스폰 근처로 두는 검증용 레벨
   변형을 호스트가 자동 생성하는 방안 검토.
2. **비전 판정의 구조화**(v1.6 권고 승계): 스크린샷 분석을 자유 서술 대신 구조화된
   JSON(항목별 pass/fail)으로.
3. **레벨 스키마 장르 확장**: `objects[].type` enum + 로더 case 추가만으로 소코반/
   탑다운 등 확장 가능하도록 설계됨 — 실제 두 번째 장르로 검증 필요.
4. **모달 워처 데몬**: 모달 자동 처리가 여전히 compile-wait/재시도/입력 시점에만
