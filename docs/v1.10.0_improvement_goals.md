# v1.10.0 개선 목표 보고서

작성일: 2026-07-23  
기준 버전: v1.9.2  
대상: `unity_local_mcp` 오케스트레이션 및 Unity 브릿지 연동

## 1. 목표

v1.10.0의 목표는 모델의 설명이나 사람의 수동 복구에 의존하지 않고, 호스트가 Unity 프로젝트의 상태를 감지·복구·검증하도록 만드는 것이다.

이번 버전의 완료 기준은 단순 성공률이 아니다.

- 작업 중 수동 복구 0회
- 네이티브 Unity 모달 0회
- 브릿지 재연결 자동 복구
- 검증 조건 누락 0건
- 실패 항목의 무한 반복 수정 방지
- 모든 결과에 재현 가능한 증거와 복구 이력 기록

단, 서로 다른 최종 산출물 경로처럼 사용자의 의도가 충돌하는 경우에는 Unity를 변경하기 전에 명확한 오류로 중단한다. 모호한 요청을 자동으로 추측해 데이터를 변경하지 않는 것이 안전한 자동화의 경계다.

## 2. v1.9.2에서 확인된 핵심 문제

| 영역 | 관찰된 문제 | 영향 |
|---|---|---|
| 요청 해석 | 생성 경로와 합격 경로가 서로 다름 | 작업 후반에 구조적으로 합격 불가 |
| 씬 저장 | dirty Untitled 씬에서 Save Scene 모달 발생 | 브릿지 중단 및 사람의 클릭 필요 |
| 브릿지 | 컴파일 후 도메인 리로드와 브릿지 재연결을 동일하게 취급 | 실제 컴파일 성공도 타임아웃 처리 |
| Unity 선택 | 첫 번째 Unity 창을 포커스 | 여러 프로젝트 실행 시 잘못된 프로젝트 조작 가능 |
| 검증 | 선행 조건 실패 후에도 입력·카메라 검증 계속 | 낙하한 Player에 대한 무의미한 측정 반복 |
| repair | builder 실패를 verifier repair가 대신 처리 | 계획 범위 이탈 및 중복 오브젝트 생성 |
| 반복 제어 | 동일 fingerprint만 비교 | 실패 수가 증가해도 계속 수정 |
| 정적 품질 | 정책 위반 코드를 Play Mode에서 발견 | 수정 비용과 검증 시간이 증가 |

## 3. 개선 원칙

1. **사전 검사 우선**: Unity 변경 전에 경로·상태·범위·정책을 검사한다.
2. **호스트 결정성**: 포커스, 재연결, 저장, 재시도는 모델의 판단이 아니라 호스트 상태 머신이 수행한다.
3. **증거 기반 진행**: 성공·실패뿐 아니라 측정값, 시점, 도구 응답을 저장한다.
4. **선행 조건 준수**: 필요한 조건이 충족되지 않으면 후속 검증을 실행하지 않고 `blocked_by`로 기록한다.
5. **최소 변경**: 성공한 산출물은 잠그고 실패한 항목에만 수정 예산을 사용한다.
6. **안전한 중단**: 자동 복구 한도를 넘거나 새 치명적 실패가 생기면 추가 변경을 멈춘다.

## 4. 주요 개선 항목

### 4.1 프로젝트·요청 preflight

`RequirementsPreflight`를 추가해 다음을 Unity 호출 전에 검사한다.

- 최종 씬 경로가 하나인지 확인
- 필수 파일과 클래스명 충돌 확인
- 금지 API·태그·패키지 요구 확인
- 현재 Unity 프로젝트 경로와 활성 씬 확인
- Play Mode, 컴파일, dirty 상태 확인
- 작업 범위와 `forbidden_scope` 확정

충돌 시 결과는 `preflight_failed`와 충돌 목록으로 남기며 mutation tool은 호출하지 않는다.

### 4.2 모달 없는 씬 트랜잭션

씬 생성·저장을 `create_scene_atomic(target_path, dirty_policy)` 작업으로 통합한다.

- `dirty_policy=backup`: Untitled 또는 dirty 씬을 복구 폴더에 자동 백업
- `dirty_policy=save`: 명시된 경로로 저장
- `dirty_policy=abort`: 사람 확인 없이 안전 중단
- 저장 후 `scenePath`와 `isDirty=false`를 호스트가 확인

`unity_save_scene {}`처럼 경로가 없는 호출은 Untitled 씬에서 허용하지 않는다.

### 4.3 Unity 포커스 및 브릿지 복구 상태 머신

상태를 다음과 같이 분리한다.

```text
READY
  → COMPILE_REQUESTED
  → DOMAIN_RELOAD
  → BRIDGE_UNAVAILABLE
  → BRIDGE_RECONNECTING
  → PROJECT_IDENTITY_CHECK
  → READY
```

필수 동작:

- 프로젝트 경로와 Unity 프로세스 명령줄을 이용한 창 선택
- 포커스 요청 후 실제 foreground 창인지 확인
- 포트 번호 변경 없이도 liveness 재확인
- 브릿지 generation 또는 세션 ID로 재시작 감지
- 단계별 timeout과 `recovery_event` 기록
- 자동 복구 횟수 초과 시에만 사람에게 알림

### 4.4 검증 DAG와 정확한 측정

검증 조건을 선형 체크리스트가 아니라 의존성 그래프로 관리한다.

```text
asset_saved
  → compile_clean
  → required_components
  → stable_spawn
  → movement
  → jump_and_landing
  → boost
  → camera_follow
  → screenshot_and_cleanup
```

각 조건은 반드시 `pass`, `fail`, `blocked`, `not_measured` 중 하나와 측정값을 가져야 한다. 요청이 1초 측정을 요구하면 검증기도 1초를 사용하며, 이동·점프·부스트 측정 전에는 동일한 시작 상태로 복원한다.

### 4.5 Builder와 repair의 책임 분리

- builder의 최소 산출물 생성 실패는 builder 재계획으로 처리
- verifier repair는 이미 생성된 산출물의 실패 항목만 수정
- repair prompt에 허용 파일·오브젝트·도구와 금지 범위를 전달
- 동일 이름 GameObject, 중복 클래스, 범위 밖 스크립트 생성을 도구 레벨에서 거부
- 마지막 mutation 이후 자동 save barrier 실행

### 4.6 정책 lint와 구조 검증

컴파일 전에 정적 검사를 수행한다.

- legacy `UnityEngine.Input` 사용 금지
- `Keyboard.current` null check 확인
- `Rigidbody.linearVelocity` 사용 확인
- `CompareTag`의 정의되지 않은 태그 사용 금지
- 필수 메서드의 실제 호출 여부 확인
- Rigidbody constraint bitmask가 요구사항을 모두 포함하는지 확인
- 클래스명과 파일명 일치 확인

### 4.7 회귀·무진전 감지

각 repair cycle에서 `resolved`, `persisted`, `new` 실패를 비교한다.

- 새 P0/P1 실패 발생 시 cycle 중단
- 두 사이클 연속 개선 없음 시 재계획
- 동일 도구 반복 비율이 임계치를 넘으면 batch/snapshot 방식으로 전환
- 성공한 조건은 보호 목록에 추가
- 모든 cycle에 변경 파일, 변경 오브젝트, 결과 fingerprint 기록

## 5. 버전별 구현 계획

### Cycle A — 자동 복구 기반

- 요청·프로젝트 preflight
- 모달 없는 씬 저장
- 프로젝트별 Unity 창 식별
- foreground 검증
- 동일 포트 브릿지 재연결
- 컴파일 상태와 브릿지 상태 분리

### Cycle B — 검증 신뢰성

- 검증 DAG
- 조건별 `pass/fail/blocked/not_measured`
- 측정 전 상태 리셋
- exact duration과 수치 기준 준수
- snapshot/batch 조회

### Cycle C — 안전한 repair

- builder/repair 분리
- `forbidden_scope` 전달
- 중복 생성 차단
- 정적 정책 lint
- 저장 barrier와 변경 예산

### Cycle D — 운영 품질

- 회귀 점수와 자동 중단
- 구조화된 복구 이벤트
- 실행별 지표 대시보드
- 버전 단일화 및 호환성 테스트

## 7. Cycle A 완료 테스트

1. dirty Untitled 씬에서 대화상자 없이 목표 씬을 생성·저장한다.
2. 세 스크립트 컴파일과 도메인 리로드 후 사람의 포커스 조작 없이 브릿지가 복구된다.
3. Unity 프로젝트 두 개가 열려 있어도 지정 프로젝트만 조작한다.
4. 브릿지가 같은 포트로 재시작되어도 세션을 재검증하고 연결한다.
5. 서로 다른 씬 경로가 요청되면 Unity mutation 없이 명확히 중단한다.
6. 각 복구 단계가 JSON 로그에 원인·시도 횟수·결과·소요 시간을 남긴다.

## 8. 예상 효과

v1.10.0은 모델의 추론 품질만 높이는 버전이 아니라, 실패를 조기에 감지하고 안전하게 복구하는 실행 기반 버전이다. 우선 Cycle A를 적용하면 이번 v1.9.2 실행에서 발생한 Save Scene 수동 클릭과 Unity 포커스 개입을 제거할 수 있고, 이후 Cycle B~D를 통해 무의미한 검증 반복과 범위 밖 repair까지 줄일 수 있다.
