# v1.9 독립 검증 오케스트레이션

현재 구현 버전: **v1.9.1**

v1.8.1까지의 `TaskContract`는 모델이 필요한 도구를 호출했는지는 강제했지만,
요청 전체가 실제로 동작하는지와 모델의 자연어 완료 보고는 완전히 분리하지 못했습니다.
v1.9는 **완료 판정권을 모델에서 호스트로 이동**합니다.

## 실행 흐름

1. 제작 모델이 단일 또는 플래너 모드로 작업합니다. 이 단계의 자연어 완료 보고는
   검증 전 임시 결과이므로 화면에 최종 답변으로 노출하지 않습니다.
2. `VerificationSpec`이 원래 요청에서 파일·씬·게임플레이·이동·점프·카메라 추종 등
   검증 항목을 만듭니다.
3. 호스트가 `verify` 도구 경계에서 Unity를 직접 조회하고 입력을 주입합니다. 검증
   순서는 로컬 모델이 결정하지 않습니다.
4. 하나라도 실패하면 측정값과 실패 코드만 새 수정 컨텍스트에 전달합니다. 수정
   컨텍스트는 성공한 산출물을 재생성하지 않고 실패 항목만 고칩니다.
5. 수정 후 이전 주장을 재사용하지 않고 새 `VerificationContract`로 전부 다시
   측정합니다.
6. 모든 항목이 통과한 경우에만 호스트가 `verified` 영수증을 만들고 완료로
   판정합니다.

```text
요청 → 제작(완료 판정 불가) → 호스트 실측
                           ├─ 통과 → verified 영수증
                           └─ 실패 → fresh 수정 → 호스트 재실측
```

## 호스트가 수집하는 증거

- 요청에 명시된 `Assets/...` 파일의 실제 존재
- 활성 씬 경로, `isDirty=false`, 최종 `isPlaying=false`
- Edit Mode 컴파일 오류/예외 0건
- 요청에서 추론한 필수 컴포넌트의 실제 부착 상태
- Play 진입, 대기 후 런타임 오류/예외 0건
- 데이터 주도 레벨 요청의 `[LevelLoader] Loaded` 로그
- `rightArrow` 입력 전후 Player X 변화량
- 같은 구간의 Main Camera X 변화량
- `space` 입력 전후 Player Y 변화량
- Play 중 생성된 Game 뷰 PNG 파일
- 가상 키의 held/pending release 비움과 Play Mode 종료

v1.9.1부터 Play Mode 진입 응답만 믿지 않고 입력 직전에 `unity_get_state`로
실제 `isPlaying=true`를 확인합니다. Play가 중간에 끝나면 입력 호출을 계속하지
않고 `play_mode_ended_unexpectedly`로 기록합니다. 요청에 실제 스크립트 경로가
포함되어 있으면 파일명에서 얻은 정확한 클래스명을 컴포넌트 검증에 사용합니다.

도구 호출 성공 자체는 동작 성공으로 간주하지 않습니다. 예를 들어
`unity_send_key`가 `status=ok`여도 Player 좌표가 변하지 않으면 검증 실패입니다.

A/D가 요청된 경우 양방향 이동을 따로 측정하며, Shift/부스트가 요청된 경우
일반 D 이동 거리와 D+Shift 이동 거리를 비교해 부스트가 1.4배 이상인지 확인합니다.
점프는 단일 시점이 아니라 두 개의 Play Mode 샘플 중 최고 Y를 사용합니다.

## 검증 영수증

영수증은 `logs/receipts/YYYY/MM/DD/*.json`에 저장되며 다음을 포함합니다.

- `status`: `verified` 또는 `failed`
- 원래 요청과 호스트가 만든 검증 명세
- 컴파일/런타임 오류 목록과 좌표 변화량
- 스크린샷 경로, 입력 해제, 최종 Play 종료 상태
- 매 검증/재검증 시도의 실패 코드
- 제작 단계 성공 여부와 전체 소요 시간

CLI의 `/receipt` 또는 `/log`로 마지막 영수증 경로를 다시 확인할 수 있습니다.

## 한계와 종료 정책

“완벽한 검증”은 요청을 기계적으로 측정 가능한 조건으로 바꿀 수 있는 범위에서만
가능합니다. 현재 휴리스틱이 모르는 고유 게임 규칙, 시각적 품질, 재미는 자동으로
증명할 수 없습니다. 따라서 v1.9는 한계를 무작정 제거하거나 무한 반복하지 않습니다.

- 기본 자동 수정은 최대 3회입니다.
- 전체 검증/수정 시간 예산은 기본 1,800초입니다.
- 같은 실패 집합이 기본 2회 반복되면 `no_verification_progress`로 종료합니다.
- 실패 종료도 영수증을 남기며 절대로 완료로 표시하지 않습니다.

고유 규칙을 엄격히 검증하려면 이후 버전에서 명시적 acceptance spec 또는 프로젝트별
검증 플러그인을 추가하는 방식이 안전합니다.
