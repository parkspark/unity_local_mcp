# v1.5: local-only orchestration and verification

This release makes the local Ollama model a proposer, not the authority that
decides whether a Unity task is complete.  `qwen3-coder:30b`,
`qwen2.5vl:32b`, the MCP stdio server, and the Unity TCP bridge all run on the
local machine; no cloud or external model API is required.

## Execution flow

```text
local coding model
  -> TaskContract policy gate
  -> local Unity MCP / Unity Editor
  -> compile and runtime verification gate
  -> optional local vision feedback
  -> coding model receives evidence and continues or reports completion
```

## What is enforced

- `unity_execute_menu_item` is blocked for agent turns.  Menu items can open
  native Windows dialogs, so they cannot be reliably orchestrated.
- A new `unity_create_scene(path, template)` tool creates and saves a scene at
  an `Assets/Scenes/*.unity` path without a Save dialog.  It refuses to discard
  dirty scenes and refuses to overwrite an existing scene.
- Script reads and deletes are limited to scripts created in the current
  session, unless the user explicitly names an `Assets/...` path in the
  request.  This prevents a task from copying unrelated sample code.
- Script writes are limited to `Assets/Scripts/*.cs`.
- `t:Script` searches are automatically scoped to `Assets/Scripts` and return
  at most 30 paths.  The underlying MCP server also defaults all asset searches
  to `Assets/`, excluding `Packages/`.
- After a script write, the agent must refresh assets and read compiler errors
  before it can finish.  After play mode, it must wait briefly, read runtime
  errors, and stop play mode.

## Local vision

The installed default is `qwen2.5vl:32b`.  Manual inspection remains available
through `uv run main.py --vision` and `/look`.

For a fully automated local feedback loop, set this before starting the CLI:

```powershell
$env:UNITY_AGENT_AUTO_VISION = "1"
uv run main.py
```

This invokes the local vision model after `unity_screenshot` and adds its
observation to the coding model's next turn.  On the 32 GB VRAM target machine
the coder and vision model should run sequentially; automatic vision is off by
default to avoid retaining both large models in memory.

## Current visual limitation

`unity_screenshot` currently renders a camera.  Legacy `OnGUI` and screen-space
overlay UI can be absent from that capture even when visible in the Unity Game
view.  Treat vision feedback for those UI modes as inconclusive.  For
machine-verifiable games, render game-critical state through camera-visible
objects (or extend the bridge with a post-GameView capture path) and retain the
runtime/console checks as separate evidence.
