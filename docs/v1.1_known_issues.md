## 1. API Updater 동의 모달 — 브리지 무기한 마비

로컬 모델이 사용 중단된 API(`Rigidbody.velocity`)로 스크립트를 작성 → `unity_refresh_assets` 호출 시
Unity가 **"Script Updating Consent"** 모달을 띄움 → 에디터 메인 스레드 정지 → 브리지 무응답.

호스트의 컴파일 대기 로직(최대 90초 폴링)은 이 케이스를 전혀 커버하지 못해 **5분 넘게 자동 복구 불가**
상태로 멈췄다. 이전에 처리한 "씬 리로드 확인" 모달과 같은 부류지만 사전 대비가 없었던 새로운 케이스.

**재현 조건**: 모델이 obsolete API를 사용하는 스크립트를 작성하고 refresh할 때마다 발생 가능.

**임시 완화**: 한 번 "Yes, for these and other files that might be found later"를 선택하면 그 에디터
세션 동안은 재발하지 않음 (확인됨).

**필요한 조치** (미착수):
- 시스템 프롬프트에 흔한 obsolete API 금지 규칙 추가 (`rb.velocity` → `rb.linearVelocity` 등)
- 또는 `EditorPrefs`/커맨드라인으로 API Updater 동의를 사전에 비활성화하는 Editor 스크립트 추가
- 또는 호스트가 컴파일 대기 타임아웃 시 "모달이 떠 있는지" 감지해 사용자에게 명확히 알리는 로직

## 2. SceneAutoReload가 dirty 씬에서는 개입하지 않음

씬에 저장 안 한 변경(새 GameObject 등)이 있을 때는 `SceneAutoReload.cs`가 의도적으로 자동 리로드를
건너뛰고 Unity 기본 프롬프트를 그대로 둔다 (작업 유실 방지를 위한 설계). 문제라기보다 **알려진 트레이드오프**—
자동 제작 세션 중에는 씬이 거의 항상 dirty 상태라 이 프롬프트를 계속 마주치게 된다.

## 3. 강제종료 시 고아 프로세스 재발

모달로 멈춘 에이전트 프로세스를 background task 강제 종료로 내렸더니, `uv.exe` 1개 + `python.exe`
(server.py) 2개가 고아로 남았다. 정상 종료(`/quit`, 정상 예외)에서는 `AsyncExitStack`이 자식 프로세스를
정리하지만, **하드 킬 경로는 커버되지 않는다.**

**필요한 조치** (미착수): MCP 서버 자식 프로세스를 Windows Job Object에 묶어, 부모가 어떻게 죽든
자동으로 함께 종료되게 만드는 것을 검토.

## 4. 검증 절차의 구조적 한계 — "콘솔 에러 없음" ≠ "정상 작동"

에이전트는 "플레이 모드 5초 실행 → 콘솔 에러 확인" 방식으로 검증했고 에러 없음을 근거로 정상 작동을
보고했다. 그러나 실제 컴포넌트 데이터를 직접 조회하니 **플레이 불가능한 상태**였다:

| 버그 | 실제 영향 |
|---|---|
| `groundLayer`가 `Nothing`(0)으로 미설정 | `Physics.CheckSphere` 항상 실패 → `isGrounded` 영구 false → 점프가 더블점프 폴백 로직 덕에 딱 1회만 되고 이후 영구 불가 |
| 이동 입력이 X축만 처리 | 플랫폼/장애물/목표가 전부 Z축(2~10)에 배치되어 있어 플레이어가 물리적으로 도달 불가능 |
| `Vector2`를 `Vector3` 필드에 대입 | 암묵 변환으로 Z속도가 매 프레임 0 강제 (위 문제와 동일 근본 원인) |

세 버그 모두 예외를 던지지 않는 **조용한 로직/설계 버그**라서 콘솔 로그 기반 검증으로는 원천적으로
탐지 불가능하다. 시스템 프롬프트의 "verify after act" 규칙이 예외 확인 수준에 머물러 있는 것이 근본 원인.

**필요한 조치** (미착수): 스크린샷 + 비전 모델(`--vision`)을 필수 검증 단계로 승격하거나, 레벨 배치 축과
컨트롤러 입력 축의 일치 여부 같은 체크리스트를 시스템 프롬프트에 명시.
